# OpenClaw 课程文章生成规则

这个文件用于固定后续课程文章的写作和配图流程，避免每次生成时标准漂移。

后续生成 OpenClaw 课程文章前，先读取本文件，再开始写作。

## 总流程

每个 Day 都按下面顺序执行：

```text
1. 先写完整中文文章
2. 再写完整英文文章
3. 根据中文文章内容提炼中文配图主题
4. 根据英文文章内容提炼英文配图主题
5. 生成两张配图：中文图 + 英文图
6. 中文文章只引用中文图
7. 英文文章只引用英文图
8. 更新《课程的目录.md》里的对应 Day 链接
```

不要先根据标题生成图片。图片必须服务文章内容。

## 文件命名

文章统一放在：

```text
articles/
```

命名格式：

```text
day-XX-英文短标题-zh.md
day-XX-英文短标题-en.md
```

示例：

```text
articles/day-03-model-tools-browser-zh.md
articles/day-03-model-tools-browser-en.md
```

配图统一放在：

```text
assets/
```

每篇文章生成两张图：

```text
assets/day-XX-英文短标题-zh.png
assets/day-XX-英文短标题-en.png
```

## 文章质量标准

中文文章和英文文章都不能只写成说明文，要写成适合发布的课程文章。

英文文章不能只做机械直译，要保留中文文章的结构和观点，同时用自然英文重写。

每篇文章应尽量包含：

- 开头钩子：先抓住读者的疑问、误解或痛点
- 明确结论：尽早告诉读者这一篇到底解决什么问题
- 概念拆解：把抽象名词拆成能理解的模块
- 流程图或结构图：用 Markdown 代码块展示请求流、执行流、架构流
- 对比例子：例如“普通 Agent vs OpenClaw”
- 真实场景：企业微信、浏览器自动化、MCP、SaaS、部署排错等
- 常见误解：专门指出读者容易理解错的地方
- 学习顺序：告诉读者下一步应该怎么学
- 本节作业：给读者 3 到 5 个可以立刻执行的小任务
- 总结收束：用几句话把核心观点钉住
- 下一节预告：说明下一节会解决什么问题，让课程形成连续感

## 写作风格

中文文章风格要求：

- 中文表达，面向新手但不降低技术密度
- 段落短，阅读节奏快
- 多用具体例子，不要空泛堆概念
- 多用“你会遇到什么问题”来展开
- 避免营销腔，避免过度夸张
- 不复制参考文章原文，保持原创表达
- 技术概念要准确，不能凭空编造

英文文章风格要求：

- Natural English, not machine-translated wording
- Keep the same core structure as the Chinese article
- Preserve technical accuracy and examples
- Use short paragraphs and clear section headings
- Avoid marketing language and exaggerated claims

推荐文章长度：

```text
5000 到 10000 中文字符
```

如果主题很小，可以略短；如果主题涉及部署、模型、插件、企业实战，可以更长。

## 技术资料要求

写具体 OpenClaw 技术概念前，优先查官方文档或已有权威资料。

尤其是下面内容，必须确认后再写：

- Gateway
- CLI 命令
- Bridge / ACP / MCP
- Workspace
- providers 配置
- Docker / docker-compose
- 插件系统
- Browser / Shell / Canvas
- 端口、认证、环境变量
- 调试排错方式

文章末尾可以放“参考资料”，链接到官方文档或对标资料。

## 配图规则

每篇文章生成两张配图：

```text
中文图：标题和主要标签使用中文，只插入中文文章
英文图：标题和主要标签使用英文，只插入英文文章
```

配图应根据文章内容生成，而不是只根据标题生成。

生成图片前，先从文章中提炼：

- 核心主题
- 关键模块
- 最重要的流程
- 读者应该记住的视觉结构

配图风格统一：

- 技术课程信息图 / 架构图
- 16:9 横图
- 白色或极浅灰背景，禁止黑色、深色、赛博风背景
- 参考用户提供的白底架构图风格：大标题、流程卡片、分区模块、箭头连接、底部总结栏
- 主色使用深蓝、亮蓝、绿色，少量橙色和紫色做分区点缀
- 模块卡片使用浅色描边和轻微阴影，整体清爽、干净、像课程讲义
- 架构感、流程感、系统感、教学感
- 内容密度可以高，但必须有清晰层级：主流程、能力层、工具层、Workspace、示例或总结
- 不要人物
- 不要吉祥物
- 不要杂乱装饰，不要抽象光效，不要深色渐变
- 标签必须尽量清晰可读

## Markdown 结构

中文文章顶部格式：

```md
# 文章标题

![中文配图](../assets/xxx-zh.png)
```

英文文章顶部格式：

```md
# Article Title

![English cover](../assets/xxx-en.png)
```

正文中优先使用：

````md
## 二级标题

### 三级标题

```text
流程图
```
````

文章末尾固定使用下面顺序：

```md
## 最后总结

## 本节作业

## 下一节预告

## 参考资料
```

如果英文文章，对应使用：

```md
## Final Summary

## Lesson Homework

## Next Lesson Preview

## References
```

目录文件中的链接格式：

```md
- [x] Day X：[中文](articles/xxx-zh.md) / [English](articles/xxx-en.md)：文章标题
```

- [ ] 英文配图已写入 `assets/`
- [ ] 中文 Markdown 顶部只插入中文图
- [ ] 英文 Markdown 顶部只插入英文图
- [ ] `课程的目录.md` 已更新链接
- [ ] 文章包含“本节作业”
- [ ] 文章包含“下一节预告”
- [ ] 文章标题层级正常
- [ ] 图片路径可用
- [ ] 如涉及具体技术，已附参考资料
